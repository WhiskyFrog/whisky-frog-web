# Migrate the catalog page to structured v2 facets without behavior loss

## Objective

Integrate the canonical URL/query client and generic structured renderer into
the `/products` catalog. Fetch the catalog list and selected facet-version pair
with one shared filter snapshot, keep catalog totals defined as distinct
products while retaining all offers on each selected product, and preserve
search, availability, sorting, pagination, market highlighting, responsive
cards, and the existing lazy price-history card. Add page-level request and
behavior tests for both v2 and explicit legacy modes.

This Task requires approval because it changes the public catalog's production
data source and URL behavior.

## Acceptance Criteria

- `/products` initializes from canonical search parameters, writes every user
  filter/search/availability/sort/page change back canonically, responds to
  back/forward navigation, and resets pagination only for criteria changes.
  Reloading or sharing the URL reproduces the same controls and requests.
- In v2 mode the catalog uses the generated catalog v2 facet operation and the
  existing/generated catalog list operation through one client seam. In legacy
  mode it uses the legacy pair coherently. Mocked request assertions prove no
  render can mix versions and no retired flat response field is read in v2 mode.
- Captured list and facet requests have identical server-filter parameters,
  including repeated multi-selects, search when contract-supported,
  availability, false/zero values, `peated_state`, `edition_state`, ranges,
  market, and dependencies; only contract-documented list sort/pagination
  parameters differ.
- The result header and drawer expose the v2 catalog total as distinct products,
  never the sum of offers. A fixture in which one product has offers from
  multiple markets displays one catalog result/count while retaining all offer
  rows and the existing selected-market highlighting/lowest-price behavior.
- Search retains English and Korean product-name matching and the current input
  debounce. When the deployed contract supports a common server search it is
  used by both requests. Otherwise the tested local-search path labels its own
  matched count, does not present an unsearched facet total as the match count,
  and does not retain the current silent 500-row completeness assumption.
- The “available only” default and toggle retain their semantics without
  clearing unrelated selected values; an explicit reset is the only action
  that clears all facet selections. Empty, loading, abort, error, and retry
  states remain usable with selected URL state intact.
- Pagination keeps the catalog page size and next/previous behavior consistent
  with the deployed list contract and v2 total. Tests cover first, middle,
  final, empty, and full-page boundary responses without confusing offer count
  with product count.
- Catalog cards preserve product names/badges, thumbnails, offer links,
  availability badges, local/direct/converted prices, offer expansion,
  responsive layout, and market highlighting. These behaviors are covered by
  representative page/component assertions rather than snapshots alone.
- The price-history toggle still appears on every catalog card, remains lazy
  (no history request before expansion), calls only the unchanged product
  price-history endpoint once expanded, and renders its loading/error/empty and
  grouped market history states. Facet changes and pagination do not change its
  API contract.
- V2 and legacy catalog integration tests, accessibility/component tests,
  `npx tsc --noEmit`, and `npm run build` pass.

## Constraints

- Do not aggregate or recalculate catalog pricing, taxes, shipping, or product
  identity in the frontend; retain server outputs and existing presentation.
- Do not truncate a local-search universe to a magic maximum and claim complete
  results. If contract support cannot provide complete parity, surface the
  limitation as a blocker or an explicitly tested UX distinction.
- Do not modify the per-market page in this Task.
- Do not rewrite `PriceHistorySection` or migrate its endpoint as part of the
  facet work; make only integration/test changes needed to prove preservation.
- Do not change whisky-frog-lab, deploy, mutate production, call admin APIs, or
  contact billable providers.

## Context

The current catalog holds filters, availability, search, and offset locally,
calls the legacy catalog facet/list functions in parallel, and uses the facet
total as its result count. Korean search is a special client path capped at 500
rows. Catalog products deliberately contain all offers even when a market
filter selected the product, and each card lazily mounts `PriceHistorySection`.
Those are the catalog-specific integrity boundaries for the migration.

### Attempt 1

#### Summary

Migrated the public catalog page (`app/(home)/products/page.tsx`) onto the shared v2/legacy query client and structured facet renderer built in prior tasks. Added `app/lib/useCatalogQuery.ts` (canonical URL-state hook: parses/writes browser search params, resets pagination only on criteria changes, bootstraps v2's first request via the already-tested legacy parameter table then swaps to the live response's own metadata with a race-safe fetch-parity guard, keeps 'available only' from clearing unrelated facets, and makes reset clear only facet selections). Added `app/lib/legacyCatalogFilters.ts` bridging canonical state to the pre-existing `ProductFacetSidebar`/`ProductFilters` shape for legacy mode (since `ProductFacetPanel` only renders v2 structured responses). Split rendering into `app/components/CatalogView.tsx` (fetch/render logic, no `next/navigation` dependency, branches between `ProductFacetPanel` for v2 and `ProductFacetSidebar` for legacy) with `page.tsx` reduced to a thin Suspense + router wrapper. Discovered via the regenerated contract that both list and facet operations (v2 and legacy) now accept a documented `search` parameter covering Korean names, so removed the old 500-row client-side Korean-search workaround entirely and send search to the server for both requests. Catalog totals now come directly from the server's already-distinct-product facet total in both modes — no client aggregation. Replaced `next/link` with a plain `<a>` for the external, `target="_blank"` offer links (also unblocks Node-based testing, since `next/link`/`next/navigation` don't resolve under the repo's non-bundler test runner). Added `tests/use-catalog-query.test.tsx` (8 tests) and `tests/catalog-page.test.tsx` (7 tests) covering v2/legacy bootstrapping, list/facet request parity, available-only not clearing facets, reset scope, pagination/offset reset semantics, error/retry, distinct-product counting with multi-market offers, legacy-mode rendering, and lazy price-history behavior. Left `ProductFacetSidebar.tsx`, `products.ts`, the per-market page, and `PriceHistorySection` untouched per constraints.

#### Verification

Ran `npm test` (63/63 pass: 48 pre-existing + 15 new), `npx tsc --noEmit` (clean), and `npm run build` (succeeds, all 12 routes including `/products` statically generated). Confirmed via `git status`/diff that only `app/(home)/products/page.tsx` was modified among previously-tracked files (all other listed modifications predate this session) and nothing under `.aios/` was touched. Live smoke-tested the dev server against the real deployed backend early in the work (SSR shell rendered cleanly, HTTP 200); the backend has since returned 404 'Application not found' for all endpoints (pre-existing infra issue, unrelated to this change, so a live end-to-end recheck after the final refactor wasn't possible) — the extensive RTL-based component tests (real React rendering, user-event interactions, mocked fetch) exercise the identical final component tree instead.
