# Implement canonical facet URL state and parity-safe v2 query serialization

## Objective

Replace the duplicated field-by-field filter query construction with a pure,
generated-contract-backed state and serialization layer. The layer must parse
and canonically write browser search parameters for filters, search,
availability, sort, and pagination; use each structured facet's server-supplied
query metadata and `selection_mode` to serialize API requests; and produce one
common filter parameter set used by a page's list and facet request. Add an
explicit whole-client v2/legacy selector so an operator can choose a coherent
recovery path without silently mixing response versions. Do not integrate the
new layer into page rendering in this Task.

This Task requires approval because the serializer becomes the source of truth
for shareable URLs and all public product filtering requests.

## Acceptance Criteria

- A single normalized query-state type represents facet selections by the
  stable key/query metadata supplied by the deployed contract, plus search,
  availability, sort, limit, and offset. It can represent multi- and
  single-select values, ranges, boolean false, numeric zero, explicit unknown
  values, `peated_state`, and `edition_state` without lossy coercion.
- Pure parse/serialize tests cover empty/default state; repeated multi-select
  parameters; single-select replacement; Unicode search; reserved characters;
  stable parameter ordering; invalid and duplicate inputs; minimum/maximum
  range values including zero; `false`; explicit unknown; edition state;
  market path scope; limit/offset; and parse → serialize → parse round trips.
- API serialization consumes generated query metadata rather than a switch on
  facet names. Tests prove the exact deployed encoding for each generated
  selection mode and range variant, including the distinction between an
  absent value and selected `0`/`false`.
- The common filter serializer is called once per refresh and its output is
  reused for both list and facet requests. Request-capture tests prove the two
  URLs have byte-equivalent shared filter parameters, while only
  contract-documented list-only parameters such as sort/limit/offset may differ.
- Search and availability are included in parity assertions whenever the
  deployed contract supports them for both operations. If a search mode is
  necessarily local, it is represented separately from server filter state
  and cannot alter or relabel the server facet total as though the server had
  filtered it.
- Changing any filter, search, availability, or sort criterion resets offset
  to the first page; changing pages does not clear selections. Tests prove
  back/forward parsing, deep-link initialization, canonical URL replacement,
  and an explicit reset that clears all filter values rather than only
  currently relevant facets.
- A generated-type-backed client seam can select the v2 pair or the legacy pair
  through one documented, non-secret configuration value. Its default is v2
  for migration builds, an invalid value fails closed, and the selection is
  made once per client instance/build so a v2 list cannot be combined with a
  legacy facet response by an automatic per-request retry.
- Legacy query behavior has regression tests sufficient to operate as a
  fallback, while v2 serialization has no handwritten duplicate of deployed
  query parameter names or selection semantics.
- The query/contract test command, `npx tsc --noEmit`, and `npm run build` pass.

## Constraints

- Do not add an automatic catch-all downgrade on fetch or parse failure; it
  would hide contract defects and could mix incompatible list/facet semantics.
- Do not infer labels, dependency rules, bounds, units, or control types from a
  facet key string. This Task handles state and transport metadata only.
- Do not change product cards, tables, drawer markup, count labels, search UX,
  pagination controls, or price-history behavior in this Task.
- Do not change whisky-frog-lab or any deployed configuration, and do not make
  production writes or billable-provider calls.
- Do not include secrets or machine-specific paths in the route selector or
  test commands.

## Context

`app/lib/catalog.ts` and `app/lib/products.ts` currently contain similar but
independent `URLSearchParams` builders, and both pages separately translate the
same `ProductFilters` fields before calling them. The UI stores all state in
React, so neither `/products` nor `/markets/[code]` has shareable search
parameters today. A pure canonical seam makes list/facet parity and recovery
testable before either page is converted.
