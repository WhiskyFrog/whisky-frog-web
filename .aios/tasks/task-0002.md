# Establish the deployed v2 facet contract and migration test baseline

## Objective

Make the deployed OpenAPI document the sole type authority for the facet
migration. Regenerate `app/lib/api/types.gen.ts` with the repository's
production generation command, identify the generated catalog and per-market
v2 facet operations and every structured response variant, and add a small
automated test foundation plus representative contract fixtures. Record a
concise in-repository migration inventory of the two public consumers, their
local/UI state, API serializers, list counterparts, URL behavior, count
semantics, search paths, pagination, drawer behavior, and the orthogonal price
history call. Preserve a representative legacy fixture as the recovery
baseline, but do not implement UI conversion in this Task.

This Task requires approval because it refreshes a generated API boundary from
a production-hosted contract and adds project-level test tooling.

## Acceptance Criteria

- `npm run gen:api:prod` completes successfully using a read-only GET of the
  deployed OpenAPI document, and the resulting checked-in
  `app/lib/api/types.gen.ts` contains generated path/operation and response
  types for both versioned structured facet scopes as well as the legacy
  routes needed for fallback. The generated file is not manually patched.
- If either v2 route, its response discriminator, or the fields needed by the
  Brief are absent from the deployed document, the Task stops with named
  contract evidence and makes no handwritten substitute interface or guessed
  route. A stale or unavailable deployed contract is a failed preflight, not a
  successful implementation.
- Contract-facing application aliases are derived from generated
  `paths`/`operations`/`components` types. No second handwritten copy of the
  v2 response union, query parameter names, discriminator literals, selection
  mode, bounds, units, or dependency shape is introduced.
- A committed migration inventory traces
  `app/(home)/products/page.tsx` → `app/lib/catalog.ts` and
  `app/(home)/markets/[code]/page.tsx` → `app/lib/products.ts`, their shared
  `ProductFacetSidebar`, the absence of current URL search-param state, both
  list request paths, availability/search/pagination flows, the catalog
  distinct-product versus per-market-offer count meanings, and the catalog
  card's lazy `PriceHistorySection` call. It explicitly excludes admin whisky
  taxonomy facets as a different contract.
- The repository has a deterministic, non-server-starting test command suitable
  for pure TypeScript and React component tests, with DOM support only where a
  component needs it. A deliberately failing sample is demonstrated before
  finalizing configuration, then the committed baseline command passes.
- Representative legacy and v2 response fixtures are minimal, anonymized,
  read-only data shaped from the deployed schema. Together the v2 fixtures
  exercise every generated discriminator variant and include server labels,
  query metadata, every selection mode, selected/relevant true and false,
  zero-count selected options, bounds, units, dependency parents,
  `peated_state` including `unknown`, and `edition_state`.
- An automated contract test type-checks the fixtures against the generated
  types and asserts the two scope-level count semantics and required metadata.
  It fails if a fixture drifts from the regenerated deployed union or if an
  expected v2 operation is no longer present.
- `npm run build`, the new baseline test command, and `npx tsc --noEmit` all
  pass after generation and test setup.

## Constraints

- Do not modify whisky-frog-lab, deploy either repository, call admin or write
  endpoints, mutate production data, or contact a billable provider. The only
  production request allowed here is the public OpenAPI GET used by generation.
- Do not copy private backend implementation or business rules into this
  public repository. Record only public contract facts needed by the client.
- Do not change the catalog page, market page, facet UI, product list behavior,
  or active route selection in this Task.
- Do not place credentials, tokens, response data containing private values,
  or machine-specific paths in source, fixtures, scripts, or documentation.
- Keep fixtures intentionally small; they prove the public schema and edge
  states rather than snapshotting a production catalog.

## Context

The current checked-in OpenAPI output exposes the legacy flat
`CatalogFacetsOut` and `MarketFacetsOut` schemas only. The frontend then defines
additional handwritten `MarketFacets`, `FacetCount`, and range/grouping shapes
in `app/lib/products.ts`. There is no existing test suite. Establishing the
real deployed union and a runnable harness first prevents the migration from
hardcoding another soon-to-drift contract.
