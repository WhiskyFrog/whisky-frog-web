# Migrate per-market pages to structured v2 facets and offer-count semantics

## Objective

Integrate the canonical URL/query client and generic structured renderer into
`/markets/[code]`. Fetch a market's list and selected facet-version pair from
one shared filter snapshot, expose the facet total as market offer rows rather
than catalog distinct products, and preserve market lookup, search,
availability, pagination, responsive card/table selection, product editing
links for authenticated users, and all existing price presentation. Add
page-level request and behavior tests for both v2 and explicit legacy modes.

This Task requires approval because it changes every public per-market page's
production facet data source and URL behavior.

## Acceptance Criteria

- Every `/markets/[code]` page treats the encoded path segment as market scope,
  initializes the remaining state from canonical search parameters, writes
  changes back canonically, handles back/forward navigation, and resets offset
  only when a filter/search/availability criterion changes.
- In v2 mode the page uses the generated per-market v2 facet operation and its
  matching list operation through one client seam. In legacy mode it uses the
  legacy pair coherently. Mocked request assertions prove the market code is
  safely encoded, versions never mix, and v2 rendering does not read flat fields.
- Captured list and facet requests have identical server-filter parameters,
  including availability, repeated selection values, false/zero,
  `peated_state`, `edition_state`, ranges, and dependency state; only
  contract-documented list pagination parameters differ.
- The result header and drawer label the v2 total as offers/list rows in the
  selected market. Fixtures prove two market URLs for the same normalized
  product count as two offers in their individual scopes, while the UI never
  substitutes the catalog distinct-product total.
- Search retains matching over canonical English name, Korean name, and market
  raw name. If the deployed contract supports common server search, both list
  and facet requests use it. Otherwise tests keep the local presentation search
  distinct from the server total and prove its documented pagination scope so
  the page does not imply it searched unloaded offers.
- “Available only” retains its default and query behavior. Toggling it or a
  dependency does not discard unrelated, zero-count, or irrelevant selections;
  only individual clear/reset actions do. Loading, abort, retry, error, and
  empty-result states keep URL selections recoverable.
- Pagination preserves the market page size and derives previous/next behavior
  from the deployed list/total contract. Tests cover first, middle, final,
  empty, and exact-full-page responses using offer totals.
- The image-rich threshold still selects the responsive card grid, while other
  markets keep the mobile cards and desktop table. Tests retain thumbnails,
  titles/raw subtitles, availability badges, local and direct/converted price
  labels, collection times, purchase links, and authenticated taxonomy edit
  links at representative narrow and wide viewport classes.
- The page uses the same generic facet component and accessibility contract as
  the catalog; there is no per-market copy of discriminator or domain rendering.
- V2 and legacy per-market integration tests, accessibility/component tests,
  `npx tsc --noEmit`, and `npm run build` pass.

## Constraints

- Do not derive catalog distinct-product counts from a per-market response or
  relabel offers as catalog products.
- Do not change market pricing/direct-purchase business logic, admin
  authentication behavior, or taxonomy editing contracts.
- Do not add a price-history card to per-market rows as part of this facet
  migration; preserve the existing catalog-only placement.
- Do not modify the catalog page in this Task.
- Do not change whisky-frog-lab, deploy, mutate production, call production
  admin/write APIs, or contact billable providers.

## Context

The current market page loads public market metadata, a legacy flat facet
response, and 100 offer rows in parallel. Search is local to the loaded rows,
pagination infers “next” from a full page, and presentation switches among an
image-rich grid, mobile cards, and a desktop table. These semantics differ
enough from the catalog's grouped-product/offers view to warrant an independent
integration and evidence boundary.

### Attempt 1

#### Summary

Migrated `/markets/[code]` onto the shared v2/legacy query client and generic facet renderer built in prior tasks. Added `app/lib/useMarketQuery.ts` (mirrors `useCatalogQuery`'s canonical URL-state/bootstrap-then-swap/parity behavior but scoped to `{kind:'market', marketCode}`, page size 100, no catalog-only sort). Added `app/components/MarketView.tsx` (`MarketPageView`, the pure fetch/render component, router-free) reusing all existing market presentation (image-rich card grid vs. mobile-card+desktop-table, price blocks, status badges, Korean-name-priority titles, taxonomy edit links — switched from `next/link` object-hrefs to plain `<a>`/string hrefs so it renders under the Node test runner, same fix task-0005 applied to the catalog) and wiring it to `ProductFacetPanel` (v2) / `ProductFacetSidebar` (legacy, via the existing `legacyCatalogFilters` bridge). `page.tsx` is now a thin Suspense+router wrapper (`useParams`/`useSearchParams`/`useRouter`) like `/products`. Discovered via the regenerated contract that both market list and facet operations (v2 and legacy) already document a shared `search` parameter (English/Korean/raw-name ILIKE), so search is sent to the server identically on both requests — no per-market local-filter workaround. The v2/legacy facet total is now surfaced as an offer count (`N개 오퍼`), never a product count; added an optional `resultNoun` prop (default `"상품"`, unused by catalog) to the shared `ProductFacetSidebar` so its legacy drawer can say `오퍼` for market scope without touching catalog's default text/tests. Added `legacyMarketFacetFixture` to `tests/fixtures/facet-responses.ts`. Added `tests/use-market-query.test.tsx` (10 tests: v2 bootstrap→live-metadata swap, market-code percent-encoding, legacy-only routing, list/facet byte-identical parity including explicit-false/zero/peated_state/edition_state/range values, available-only not clearing facets, reset scope, pagination/offset-reset semantics, error/retry, invalid-version fail-closed) and `tests/market-page.test.tsx` (15 tests: v2 offers-not-products labeling, two market-scoped URLs counting independently as offers, market-code encoding in real requests, legacy-mode routing/labeling, available-only+facet retention, server-search parity, 5 pagination boundary cases, empty/error/retry-with-selections-intact, image-rich responsive grid, non-image-rich mobile-card+desktop-table with authenticated taxonomy edit links, Korean-name title priority). Left `app/(home)/products/page.tsx`, `app/lib/catalog.ts`, `app/lib/useCatalogQuery.ts`, `app/components/CatalogView.tsx`, and `PriceHistorySection` untouched per constraints.
