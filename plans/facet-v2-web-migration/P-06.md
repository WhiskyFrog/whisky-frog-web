# Prove migration integrity and establish the legacy removal gate

## Objective

Create and run the joined verification matrix for the catalog and per-market
facet migration, including generated-contract checks, query serialization,
component/accessibility behavior, request parity, production build behavior,
and read-only production smoke scenarios. Exercise the explicit legacy mode as
a real whole-client recovery drill, document an operator-safe rollout/fallback
procedure, and define objective evidence plus separate approval required before
legacy code can be removed. Do not deploy or remove the fallback in this Task.

This Task requires approval because its evidence and runbook define the release
and rollback boundary for a public API migration.

## Acceptance Criteria

- One documented verification command (or deterministic command group) runs
  the generated OpenAPI contract assertions, query parse/serialization tests,
  reducer/component tests, accessibility assertions, catalog integration tests,
  and per-market integration tests without starting a persistent server or
  contacting a billable provider; the complete run passes.
- A request-parity suite captures representative catalog and per-market list
  plus facet requests and proves shared parameters are equal for default,
  multi-select, single-select, zero/false, ranges, dependency parents,
  selected-irrelevant, peated unknown, edition state, search, availability, and
  paginated cases. It separately asserts distinct-product and offer-count scope.
- Browser-level production-build smoke automation covers `/products` and at
  least one configured `/markets/[code]` route at narrow and wide viewports:
  load, search, toggle availability, open/close the drawer by mouse and
  keyboard, set every structured control family, retain zero-count/irrelevant
  selections, navigate pages, reload/deep-link/back/forward, clear one value,
  reset all values, and recover from a mocked facet error.
- The catalog smoke scenario also expands an offer list and proves the
  price-history card remains lazy and renders after expansion. The per-market
  smoke scenario proves its responsive card/table mode and offer-total wording.
- A read-only production smoke script accepts explicit public base URLs and
  representative market code as arguments/configuration, performs only GET/page
  navigation, validates the deployed v2 OpenAPI signature and both list/facet
  pairs, and emits a redacted pass/fail report suitable for attaching to Task
  verification. Missing arguments, unexpected schema/version, non-2xx results,
  count/parity failures, or inaccessible controls fail closed.
- The production smoke procedure is executed against the deployed backend
  contract without authentication or data mutation. Frontend post-deployment
  smoke is documented as an operator gate and is not simulated or claimed if
  no v2 frontend release exists yet.
- Two production builds and the same critical smoke scenarios pass with the
  route selector forced to v2 and legacy respectively. The legacy drill proves
  the entire list/facet pair changes together and records the exact reversible
  configuration action; no automatic mixed-version retry is introduced.
- The rollout runbook requires: passing full tests/build, passing read-only v2
  backend smoke, operator-approved frontend rollout, passing post-deployment
  catalog/per-market smoke, and a recorded fallback drill. On failure it directs
  the operator to select the legacy pair and redeploy through the normal
  operator process; it never directs database changes or backend edits.
- The legacy-removal gate requires a separately approved change after at least
  one successfully observed v2 release boundary, repeated production smoke for
  both scopes and responsive/keyboard paths, no unresolved parity/count/search
  defects, and confirmation that the backend still exposes the expected v2
  contract. Until all evidence exists, legacy clients, fixtures, tests, and the
  selector remain intact.
- `npm run build`, `npx tsc --noEmit`, the full automated suite, and both
  configuration-specific verification runs pass, with commands and outcomes
  recorded in the Task's Implementer Verification.

## Constraints

- Do not deploy, change production configuration, remove legacy code/routes,
  or claim post-deployment evidence that was not actually observed. Those are
  separate operator-approved actions.
- Do not modify whisky-frog-lab or production data. Smoke automation must not
  authenticate, submit forms, call admin/write methods, or contact billable
  providers.
- Do not make success depend on credentials, secrets, user-local paths, or a
  specific provider/model. Public base URLs and market code are explicit
  operator inputs only.
- Do not use an automatic network-error fallback that could combine v2 list
  state with legacy facet state or conceal contract drift.
- Do not weaken or delete earlier contract, serialization, component, parity,
  count, search, pagination, accessibility, or price-history assertions to make
  the joined matrix pass.

## Context

The repository initially has no automated tests, and its legacy behavior spans
two pages with different count identities and presentation. Unit tests alone
cannot prove URL navigation, responsive drawer focus, request pairing, or the
lazy price-history card. Conversely, a manual smoke alone cannot prove lossless
zero/false serialization or generated-union exhaustiveness. The final boundary
therefore joins layered automated evidence and preserves an explicitly tested
recovery mode until a later approval authorizes removal.
